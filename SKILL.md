# 二期数据质检 Skill

> 定位：**严格初步质检**，快速筛出明显违规数据。关键词+正则为主，不做语义理解。宁可误报，不可漏报。

---

## Phase 0: 启动质检（3 种方式选一）

| 方式 | 适用场景 | 操作 |
|------|---------|------|
| **A. 桌面 .bat** | 本地运行，无需公网 | 双击桌面「运行质检.bat」 |
| **B. 飞书按钮** | 单条记录触发 | `/rec/<record_id>` 接口，飞书自动化按钮 |
| **C. 批量 API** | 全表检查 | `POST /test/check` + `{"url": "表格链接"}` |

**前置条件**：
1. 服务已启动：`cd feishu-qa-bot && python server.py`
2. `.env` 已配置 `FEISHU_APP_ID` 和 `FEISHU_APP_SECRET`
3. 飞书应用已开通 `bitable:app` 权限，且表格已将应用加为协作者（可编辑）

---

## Phase 1: 执行质检

质检引擎自动执行以下流程：

```
读取飞书表格 → 自动检测字段映射 → 逐条执行 7 条规则 → 写回「是否通过」「质检备注」
```

**输出字段**（自动创建到表格末尾）：
- **是否通过**：`✅通过` / `❌不通过`
- **质检备注**：具体问题 + 修改建议

---

## Phase 2: 7 条质检规则详解

### 规则 1：场景真实性 🔴 严重

**触发条件**：题目包含用户无法操作的场景

| 模式 | 关键词 | 说明 |
|------|--------|------|
| 后台/内部 | 后台数据、后台系统、管理后台、内部系统、数据库直接、直连数据库 | 普通用户无后台权限 |
| API/接口 | API接口、调用接口、接口获取、接口数据 | 普通用户不调 API |
| 爬虫/抓取 | 爬虫、爬取、抓取数据、scrape、crawl | 非普通用户操作 |
| 非法操作 | 黑入、破解、入侵、攻击、绕过验证、破解密码 | 违法+不真实 |
| 不合理限制 | 不使用.*搜索、不用.*APP、不通过.*工具.*查询 | 限制正常工具却要求专业数据 |

**不违规**：「帮我用Python写个脚本」「帮我分析这些数据」「帮我写一个API」→ 开发/分析任务合理

---

### 规则 2：精简附件描述 🟡 中等

**触发条件**：题目中透露了附件的文件名、格式、行数、时间等细节

| 类别 | 模式 |
|------|------|
| 文件名+格式 | `附件[1-9]（*.xlsx）` |
| 扩展名 | `.xlsx/.xls/.csv/.docx/.doc/.pdf/.pptx/.ppt` |
| 数据量 | `共*行、总计*条、约*行` |
| 时间范围 | `2024年1-6月、2024-01` |
| 内容概括 | `（包含*）、（共计*）` |

**命中 ≥2 个才报错**（避免单关键词误报）

**不违规**：「现在有几个附件需要你进行关联」「请参考附件中的数据」

---

### 规则 3：精简产物要求 🟡 中等

**触发条件**：产物要求包含次级格式/来源/字数限制

| 类别 | 模式 |
|------|------|
| 格式细节 | 配色、字体大小/颜色/类型、行距、页边距 |
| 数据来源 | 数据来源、注明来源、标注出处 |
| 字数限制 | 不少于X字、不超过X字、控制在X字 |
| 刚性要求 | 必须按照格式、严格按照格式、不得修改 |

**命中 ≥2 个才报错**

**不违规**：「输出Excel表格」「请用中文回答」

---

### 规则 4：题目不赘述 🟢 轻微

**触发条件**：题目 > 500 字 或 > 10 句

---

### 规则 5：模型自检不强制 🟢 轻微

**触发条件**：题目含「自检/自行检查/验证是否」且产物要求 < 20 字

---

### 规则 6：打分 checklist 必填 🔴 严重

| 状态 | 触发条件 | 严重度 |
|------|---------|--------|
| 未填写 | checklist 为空 | 🔴 |
| 过短 | checklist < 10 字 | 🟡 |

---

### 规则 7：不写 L1-3 层级 🟡 中等

**触发条件**：题目含 `L1/L2/L3`、`层级1/层级二`、`任务层级`、`任务级别1`

## 项目架构

```
feishu-qa-bot/
├── rules.yaml          ← 规则配置（改这里即可，无需改代码）
├── qa_checker.py       ← 质检引擎（从 rules.yaml 读取规则）
├── feishu_client.py    ← 飞书 API 客户端
├── server.py           ← Flask Web 服务（含所有端点）
├── .env                ← 飞书应用凭证
└── requirements.txt    ← Python 依赖
```

### 修改规则（无需改代码）

🔴 CHECKPOINT：修改 rules.yaml 前先确认当前服务状态

```bash
# Step 1: 编辑规则
# 文件位置: feishu-qa-bot/rules.yaml

# Step 2: 验证 YAML 语法
python -c "import yaml; yaml.safe_load(open('rules.yaml'))"

# Step 3: 重启生效
# 本地：重启 python server.py
# Railway：git add -A && git commit -m "update rules" && git push
```

**rules.yaml 可调参数**：

```yaml
thresholds:
  rule2_min_hits: 2        # 附件描述命中几个才报错（默认2，调到3减少误报）
  rule3_min_hits: 2        # 产物要求命中几个才报错（默认2，调到3减少误报）
  rule4_max_chars: 500     # 题目最大字数（默认500）
  rule4_max_sentences: 10  # 题目最大句子数（默认10）
  rule5_output_min_chars: 20  # 产物要求最少字数（默认20）
  rule6_min_chars: 10      # checklist最少字数（默认10）

# 加新关键词示例：
rule1:
  critical_keywords:
    - "新增关键词"          # 直接在列表末尾加
```

## 故障排除（if-then fallback 表）

| # | 症状 | 触发条件 | 一线修复 | 仍失败兜底 |
|---|------|---------|---------|-----------|
| 1 | 99991672 错误 | 读写表格报权限错误 | 飞书开放平台确认 `bitable:app` 权限已开通，且**已发布新版本** | 检查表格是否将应用/机器人加为协作者（可编辑） |
| 2 | 91403 Forbidden | 能读不能写 | 表格分享 → 找到机器人 → 权限改为「可编辑」 | 确认应用版本包含 bitable:app（不只是 readonly） |
| 3 | 字段映射不对 | 质检结果写到错误列 | 检查表格字段名是否包含「题目」「附件」「产物」「checklist」关键词 | 手动在 qa_checker.py 的 field_mapping 中指定映射 |
| 4 | 服务启动失败 | python server.py 报错 | `pip install -r requirements.txt` 安装依赖 | 检查 .env 中 FEISHU_APP_ID 和 FEISHU_APP_SECRET 是否正确 |
| 5 | Railway 部署失败 | git push 后部署报错 | 检查 requirements.txt 是否包含所有依赖 | 查看 Railway 日志：`railway logs` |
| 6 | 按钮触发无响应 | 飞书自动化按钮点了没反应 | 确认 Railway 服务在线（访问 /health） | 检查飞书自动化 HTTP 请求配置：URL 完整、Content-Type 在 Headers 里 |
| 7 | 批量更新超 500 条 | 大表格写回失败 | 代码已内置分批（500条/批），正常不会触发 | 检查飞书 API 是否有 rate limit，加 time.sleep(1) 间隔 |
| 8 | 单条质检找不到记录 | /rec/record_id 返回 404 | 确认 record_id 正确（从表格 URL 或 API 获取） | 换用 /qa/uid 接口，用 UID 列的值查找 |

## 反例黑名单（不要做的事）

| # | 反模式 | 为什么不要做 | 正确做法 |
|---|--------|-------------|---------|
| 1 | 改 qa_checker.py 来加规则 | 代码和配置耦合，维护成本高 | 改 rules.yaml，引擎自动读取 |
| 2 | 只开通权限不加协作者 | 99991672 错误，最常见的坑 | 权限 + 协作者 + 发布新版本，三步缺一不可 |
| 3 | 用「可查看」权限加协作者 | 能读不能写（91403） | 必须选「可编辑」 |
| 4 | 加了权限不发布新版本 | 权限不生效 | 开放平台 → 版本管理 → 创建版本 → 申请发布 |
| 5 | 飞书自动化 Content-Type 放 Params | 请求被拒 | Content-Type 必须在 Headers 里 |
| 6 | 规则阈值设太低 | 大量误报，用户失去信任 | 先跑一轮看误报率，再调 rules.yaml 的 thresholds |
| 7 | 单条质检用批量接口 | 浪费 API 调用，慢 | 用 /rec/record_id 只检查当前行 |
| 8 | 静默跳过字段映射失败 | 数据写到错误列或不写 | auto_detect_fields 找不到映射时应报错提醒 |
| 9 | 在 SKILL.md 里放完整正则 | 正则更新时 SKILL.md 和 rules.yaml 不同步 | 正则只放 rules.yaml，SKILL.md 只放关键词示例 |

## 严重程度说明

- 🔴 **严重**：必须修改，否则数据被拒收
- 🟡 **中等**：建议修改，影响数据质量
- 🟢 **轻微**：可选修改，优化建议
